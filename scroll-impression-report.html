isScrollInProgress 가 드래그·플링 사이에 잠깐 false 가 되는 구간

한 번 세게 스크롤하면 isScrollInProgress 는 true → false → true → false 로 기록된다. 드래그가 끝나고 플링(관성 스크롤)이 시작되기 전, 잠깐 false 가 되는 구간을 관찰하고 원인·영향·대응을 정리한 기록.

isScrollInProgress · 한 번 세게 스크롤할 때

중간의 붉은 구간 = 실제 약 2ms(한 프레임보다 짧음). 보이라고 폭을 과장했다.

true · 스크롤 동작 중 false · 드래그·플링 사이 시간 진행

배경 — 두 가지 노출(impression) 처리 방식

impressionCapturable · impressionable — 내부 동작과 차이

이 프로젝트에는 노출 이벤트를 처리하는 방식이 두 가지 있다. 역할 범위가 다르다.

impressionCapturable · 기찬님

노출 여부만 판단 → 공유 캐시에 등록/삭제

요소가 배치될 때마다(onGloballyPositioned) 화면에 보이는지 확인한다.
· 보이기 시작 → captureEvents(캐시에 추가)
· 사라짐 → releaseEvents(캐시에서 제거)
발송은 하지 않고 "지금 보이는 것 목록"만 캐시에 유지한다. 실제 발송은 별도 ImpressionEventEffects 가 스크롤이 멈출 때 캐시를 읽어 처리한다.

가시성 기준: 호스트 View 의 실제 보이는 영역(getLocalVisibleRect)

impressionable · 남표님

노출 + 스크롤 + 발송을 한 노드에서 모두 처리

요소가 배치될 때마다 보임 여부(isInViewport)를 갱신하고,
동시에 스크롤 조건(정지 또는 저속)을 관찰한다.
둘 다 충족되면 onImpression() 을 직접 호출해 이벤트를 발송한다. 스크롤 조건은 요소마다 자체적으로 가지며, 정지 판정 시 짧은 false 는 시간 지연으로 흡수한다.

가시성 기준: 호출자가 넘긴 viewportRect

getLocalVisibleRect 가 주는 영역

view.getLocalVisibleRect(r) 는 이 View 가 화면에 "보이는 부분" 을 r 에 채워준다. 좌표는 View 자신의 로컬 좌표(왼쪽 위 = (0,0), 오른쪽 아래 = (너비, 높이))다.

부모 뷰·윈도우·스크롤에 의해 잘리고 남은 사각형이 결과다:
· 전부 보이면 → (0, 0, 너비, 높이)
· 위가 잘리면 → top 이 0 보다 커짐
· 아래가 잘리면 → bottom 이 높이보다 작아짐
· 완전히 화면 밖 → false 반환

잘림 (화면 밖)
보이는 부분
= 결과 rect
← View 상단 · local (0, 0)
← 화면 상단 · local top = a
← View 하단 · local (너비, 높이)

여기서 View 는 Compose 를 담은 호스트(ComposeView)다. 그 "보이는 영역"을 얻은 뒤 getLocationInWindow 로 window 좌표로 바꿔, 각 아이템 영역과 교집합해 노출 비율을 계산한다.

주의 — z축 가려짐은 계산하지 않는다. getLocalVisibleRect 는 부모·윈도우·스크롤에 의한 기하학적 잘림만 본다. 위에 겹쳐 그려진 뷰(바텀시트·다이얼로그·오버레이·스낵바·FAB 등)로 가려져도 여전히 "보인다"고 판정한다 — 무엇이 위에 덮였는지는 보지 않기 때문. 그래서 오버레이에 가려진 아이템도 노출로 잡힐 수 있다.

구분impressionCapturable (기찬님)impressionable (남표님)
역할노출 여부 → 캐시 등록/삭제노출 + 스크롤 + 발송
발송 주체별도 ImpressionEventEffects (스크롤 트리거)자기 자신 (onImpression)
스크롤 판단없음 — 외부가 담당내장 (정지 / 저속 조건)
가시성 기준호스트 View 의 실제 보이는 영역호출자가 넘긴 viewportRect
정지 false 처리외부 스크롤 트리거에 의존내부에서 시간 지연으로 흡수
처리 단위리스트 공유 캐시요소별

→ 상품상세·상품리스트는 impressionCapturable + ImpressionEventEffects 조합을 쓴다. 그래서 발송 타이밍(아래)이 ImpressionEventEffects 의 스크롤 트리거에 달려 있고, 이 문서의 false 구간 문제도 거기서 드러났다.

원인 — scroll 세션 구조

드래그와 플링은 별개의 scroll 세션이다

핵심 전제: isScrollInProgress 는 “지금 scroll {} 블록 안에 있나” 일 뿐이다. 블록에 들어가면 true, 빠져나오면 false. 

// DefaultScrollableState — isScrollInProgress 의 정체
override suspend fun scroll(priority, block) = coroutineScope {
    scrollMutex.mutateWith(scrollScope, priority) {
        isScrollingState.value = true        // 블록 진입
        try { block() } finally {
            isScrollingState.value = false   // 블록 종료
        }
    }
}
override val isScrollInProgress get() = isScrollingState.value
01

드래그 true

손가락이 끄는 동안 scroll 세션 #1 이 열려 있다. 우선순위는 UserInput.

// ScrollableNode.drag — 세션 #1
scroll(priority = MutatePriority.UserInput) {
    forEachDelta { scrollByWithOverscroll(...) }
}
02

손 뗌 → 블록 종료 false

드래그가 끝나면 scroll {} 블록이 종료되며 finally 가 isScrolling = false 로 전환한다. false 가 시작되는 지점이다.

03

플링은 launch {} 로 예약 false 유지

남은 속도로 플링을 새 코루틴으로 시작(launch) 한다. 드래그 종료와 플링 시작이 한 번에 이어지지 않아, 그 사이가 곧 false 구간이다.

// ScrollableNode.onDragStopped — false 구간의 원인
override fun onDragStopped(event) {
    nestedScrollDispatcher.coroutineScope.launch {  // 새 코루틴(비동기)
        scrollLogic.onScrollStopped(event.velocity, ...)
    }
}
04

플링 true

scroll 세션 #2 가 열리며 다시 true. 우선순위는 Default — 드래그와 완전히 다른 세션이다. 감속이 끝나면 블록이 닫혀 마지막 false(진짜 정지).

// ScrollingLogic.doFlingAnimation — 세션 #2
scroll(priority = MutatePriority.Default) {
    with(flingBehavior) { performFling(...) }
}

관찰 A — 디스패처

collect 디스패처에 따라 false 를 관찰하는지가 달라진다

이 false 는 약 2ms 로 한 프레임보다 짧다. 그래서 같은 snapshotFlow { isScrollInProgress } 라도, 어느 디스패처에서 collect 하느냐에 따라 이 짧은 false 를 잡기도 하고 놓치기도 한다.

Dispatchers.Main.immediate

= lifecycleScope · asFlow 가 도는 곳

snapshot 적용 시점에 동기로 즉시 재read → 2ms 의 false 를 관찰함.

읽힘: true · false · true · false ← false 잡힘

AndroidUiDispatcher

= LaunchedEffect (프레임 정렬)

재read 를 프레임 경계로 미룸 → false·true 가 한 프레임에 합쳐져 최종값만 읽음.

읽힘: true ┄┄ false ← false 안 보임 (단, false 가 1프레임보다 짧을 때만)

→ 단, 이건 false 가 한 프레임 안에 들어올 때만 통한다. 프레임이 길어지면(아래) false 가 한 프레임을 넘겨 프레임 정렬이어도 그대로 보인다 — 즉 디스패처는 근본 해법이 아니다.

관찰 B — 프레임이 길어질 때

false 길이는 메인 스레드 점유에 비례하며, 디스패처와 무관하다

플링 코루틴은 launch 로 메인 스레드 큐에 예약될 뿐 곧장 실행되지 않는다. 앞선 프레임 작업(recompose · measure · 진입 레이아웃)이 길어지면 그만큼 늦게 실행되고, 그동안 isScrollInProgress 는 false 로 남는다 — false 길이 = 플링 코루틴이 큐에서 기다린 시간.

그래서 false 길이는 메인 스레드 점유율 에 비례한다. 프레임 정렬이 합칠 수 있는 건 한 프레임 안에 끝나는 false 뿐이라, 프레임이 길어져 false 가 한 프레임을 넘기면 immediate 든 프레임 정렬이든 그대로 노출된다.

false ~1ms (한 프레임보다 짧음) → 합쳐짐 / 무해

무거운 화면

조사 기록

관찰 순서

01

구조 파악

impressionCapturable 는 캐시 등록만, 발송은 ImpressionEventEffects(스크롤 트리거)가 담당. → 그 트리거가 잠깐 false 인 순간에 노출됨.

02

가설: 디스패처 차이?

asFlow 로그는 true·false·true·false (false 보임), 새로 단 LaunchedEffect 로거는 깨끗. → "collect 디스패처가 원인?"

03

같은 listState 확인

stateId 동일 → 화면·스크롤러 차이가 아님. 관찰 구조 차이로 좁힘.

04

immediate vs 프레임 정렬

Main.immediate(lifecycleScope=asFlow) 는 동기 즉시 재read → false 포착. AndroidUiDispatcher(LaunchedEffect) 는 프레임 경계로 미뤄 합침. — 디스패처 가설 성립하는 듯.

05

반증: 프레임 정렬도 발생

impressionable 은 프레임 정렬(Modifier.Node.coroutineScope)인데도 false 발생. → 디스패처만으론 설명 안 됨.

06

false 길이가 화면마다 일정

가로 섹션 ~1ms vs 상품리스트 27~79ms. 매번 달라지는 게 아니라 화면별로 일정 → 부하 의존 의심.

07

프레임 로거

상품리스트에 22·49·99ms 긴 프레임이 false 구간과 시간상 겹침. → "긴 프레임이 플링 시작을 지연".

08

Perfetto 실측

상품리스트 animation 37ms(recompose) · 상품상세 진입 measure 78ms. + JIT compiling 도배(debug 빌드).

09

결론

false 길이 = 플링 코루틴의 큐 대기시간 = 메인 점유에 비례. 디스패처는 한 프레임보다 짧은 false 만 가릴 뿐 근본 해법이 아니다.

결론

  • 이 false 는 버그가 아니다. isScrollInProgress 는 “scroll 블록 안인가”만 뜻하고, 드래그→플링 연속성을 보장한 적이 없다.
  • 원인은 두 개의 세션. 드래그(UserInput)와 플링(Default)은 별개 scroll {} 이고, 사이를 launch {} 로 비동기 인계한다.
  • RecyclerView 엔 없다. 거긴 DRAGGING → SETTLING → IDLE 한 상태머신이라 중간 IDLE 이 끼지 않는다. Compose 는 boolean 하나뿐.
  • 길이는 메인 점유에 비례. false 길이 = 플링 코루틴의 메인 큐 대기시간. 부드러우면 ~1ms(무해), 무거우면 27~79ms.
  • 디스패처로 못 막는다. 프레임 정렬은 한 프레임보다 짧은 false 만 가린다. 프레임이 길어져 한 프레임을 넘기면 immediate·frame 무관하게 샌다.
  • 확실히 막으려면 debounce / velocity. 단 debounce 는 그 시간보다 짧은 정지를 삼키고, 프레임이 매우 길면 뚫린다. 임프레션이면 onVisibilityChanged(minDurationMs) 가 더 견고.

⚠︎ 위 동작은 공식 문서의 계약이 아니라 구현 디테일(소스: Scrollable.kt · ScrollableState.kt). Compose 버전에 따라 타이밍이 달라질 수 있다.
